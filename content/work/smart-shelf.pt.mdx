## Visão Geral

Smart Shelf é um web app full-stack de acompanhamento de leitura, inspirado no Goodreads. Foi construído para praticar arquitetura real, integração de backend, consumo de APIs e dados autenticados de usuário.

O projeto permite que usuários descubram novos livros, organizem suas estantes pessoais e acompanhem sua jornada de leitura de forma fluida.

## Funcionalidades

O produto foi desenhado com forte foco em experiência do usuário e dados consistentes:

- 🔍 **Busca de livros** — Busca em tempo real com parâmetros de URL, links compartilháveis e ordenação no servidor.
- 📚 **Estante pessoal** — Salve livros com status de Lendo, Salvo ou Lido.
- 📝 **Notas** — Escreva e persista notas personalizadas por livro.
- 📊 **Estatísticas de leitura** — Acompanhe sua meta anual e seu progresso de leitura.
- 🏆 **Conquistas** — Desbloqueie badges com base na sua atividade real de leitura.
- 🔐 **Autenticação** — Cadastro por email/senha com um fluxo de onboarding em múltiplas etapas.
- 🌙 **Modo claro/escuro** — Preferência visual persistida entre sessões.

<Callout label="Fonte dos dados">
Todos os dados de livros vêm da [Open Library API](https://openlibrary.org/developers/api), um catálogo livre e aberto mantido pelo Internet Archive.
</Callout>

O projeto segue uma separação clara de responsabilidades, estruturada em três camadas principais:

- **`src/lib/`** — Clientes de API e consultas ao banco de dados (`openLibrary.js`, `userBooks.js`, `supabase.js`).
- **`src/hooks/`** — Toda a lógica de estado global e reativo (`useBookSearch`, `useAuth`, `useProfile`, `useUserBook`).
- **`src/pages/` + `src/components/`** — Camada de UI pura, mantida totalmente livre de lógica de negócio complexa.

Para otimizar a performance, os resultados de busca são cacheados em memória para minimizar a sobrecarga de chamadas externas à API. Além disso, os metadados dos livros são cacheados no Supabase assim que salvos na estante do usuário, evitando requisições repetidas a terceiros.

## Desafios & Aprendizados

Como meu primeiro projeto full-stack publicado, lidar com o desenvolvimento de ponta a ponta me apresentou a problemas reais de infraestrutura:

**Permissões do Postgres & RLS do Supabase:** Durante o desenvolvimento, consultas de usuários autenticados retornavam um erro `403 Forbidden` mesmo enviando tokens de autenticação válidos. Descobri que o problema vinha da falta de permissões `GRANT` explícitas na role `authenticated` do banco de dados, que opera junto com as políticas de Row Level Security (RLS). Resolver isso me deu uma base sólida em segurança no nível de banco de dados.

**Migração de API no meio do projeto:** Inicialmente integrei a Google Books API. No entanto, ela retornava uma quantidade avassaladora de resultados de baixa qualidade (como documentos legais e artigos acadêmicos) e não tinha um sinal de popularidade confiável. Migrei para a **Open Library API**, aproveitando a métrica `readinglog_count` e parâmetros nativos de ordenação, o que melhorou drasticamente a relevância da busca e a qualidade geral do produto.

> IA foi utilizada ao longo do projeto como parceira de pair programming para decisões arquiteturais e revisões de código. No entanto, todas as decisões de produto, design de UI/UX e a implementação final foram feitas de forma independente.
